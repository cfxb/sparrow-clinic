# Site audit — Sparrow Neuropsychology — 2026-05-14

Scope: full audit and prioritised change list against two objectives, in priority order:
1. Citation / surfacing by LLMs (ChatGPT, Claude, Gemini, Perplexity) for neuropsychology questions in or near BC.
2. Google ranking for the same query clusters.

Diagnostic context (treated as directional, per user): author-identity queries surface; condition+geography surfaces mid-list; service-finding fails; informational queries cite no clinicians. Audit weights effort toward author identity, local presence, and schema infrastructure — not toward rewriting already-strong FAQ prose.

No site content has been changed in this turn. One config-only change was made: `audit/` added to `_config.yml`'s exclude list so this internal report is not published as a public page.

### C. Author / E-E-A-T review

The single strongest signal LLMs already surface, per the diagnostic. Worth reinforcing.

**Strengths:**
- Single canonical author page (`/about/`).
- Visible bio with concrete, verifiable affiliations: VGH (with link to the specific epilepsy program page), Yale (with link to faculty profile), UCLA, Harvard, University of Melbourne, INS Epilepsy SIG (co-chair to 2026, with link), ILAE Neuropsychology of Epilepsy training (2024, 2026), Google Scholar, ResearchGate.
- Licence numbers stated: BC #4057, CT #8423.
- Bio language is consistent across pages (no contradictory framing).

**Gaps:**
- **No `schema.org/Person` on the about page** (only the thin FAQ-side reference exists). LLMs and Google have no machine-readable claim binding the entity to the external authoritative sources.
- **No `sameAs` graph** linking the site's author to: Yale faculty profile, VGH program page, Google Scholar, ResearchGate, APPCN profile, CHCPBC public register entry, LinkedIn (if applicable), ORCID (none currently linked from the site).
- **No author byline or `schema:author` on the 11 resource pages and FAQ body**, despite these being the substantive content. Resource pages should show authorship and a "reviewed by" date — both are E-E-A-T signals YMYL pages benefit from.
- **NAP not surfaced in `<head>` schema anywhere.** Address, phone, email currently appear only in the visible footer (via `_data/contact.yml`) and on the contact page.
- **No public licence-register cross-link.** The licence numbers are stated as text but not linked to the College of Health and Care Professionals of BC (CHCPBC) public register. A direct link to the register entry is one of the strongest external-trust signals available.

### E. SEO review

**Title tags** — every page has a double-brand suffix because `jekyll-seo-tag` appends `| {{ site.title }}` and page titles already contain `— Sparrow Neuropsychology`. Examples:

- `About — Sparrow Neuropsychology | Sparrow Neuropsychology` (57)
- `Epilepsy Resources — Sparrow Neuropsychology | Sparrow Neuropsychology` (70)
- `Frequently Asked Questions — Sparrow Neuropsychology, Vancouver | Sparrow Neuropsychology` (89 — well over Google's ~60-char display cap)

Three resource pages (`Encephalitis…`, `Depression and Anxiety…`, `Traumatic Brain Injury…`) and FAQ exceed 60 characters in the meaningful part before "| Sparrow".

**Meta descriptions** — **the single largest on-page SEO defect.** 19 of 20 pages share the identical 250-character site-default description set in `_config.yml`. Only `/faq/` has a unique description (152 chars, query-anchored, good). Identical meta descriptions waste a per-page SERP/LLM-snippet opportunity on every other page.

**Heading hierarchy** — every page has exactly one `<h1>`. H2s render correctly; no skipped levels detected. ✓

**Internal linking** —
- Hub-and-spoke: `/resources/` indexes 11 condition pages; condition pages link back to `/book/`; no condition→condition cross-links. Acceptable, but no horizontal linking between related conditions (e.g., MS ↔ stroke ↔ TBI all touch motor/cognitive overlap; ADHD ↔ depression-and-anxiety overlap; epilepsy ↔ TLE in FAQ).
- `/about/` is the author authority page but isn't linked from individual resource pages as the "About the author" line — only `/faq/` does this. Resource pages should each link to `/about/` from an author block.
- `/services/` is reachable from nav and from `/`, but the condition-specific pages don't link to specific anchors within `/services/`.
- Anchor text quality is mostly good — descriptive (`free 15-minute introductory consultation`, `contact page`, `Yale faculty profile`). No `click here` style.

**Cannibalisation risk**:
- `/services/` and `/resources/<condition>/` overlap on "What is X / how does neuropsychology help with X." On `/services/`, the "Why might I be referred?" bullet list names the same conditions that have full resource pages. Low risk of cannibalisation in practice because the conditions get distinct URLs, but the bullet list on `/services/` doesn't link to those resource pages — a missed internal-linking opportunity rather than a cannibalisation issue.
- `/` and `/about/` both lead with substantively the same author intro paragraph (different wording, same content). Not cannibalisation; both pages serve different intents.
- The new "Adult epilepsy" specialty card on `/` and the `/resources/epilepsy/` page do not duplicate content; the card is a pointer. ✓

**Canonical tags** — all present, but inconsistent: 19 are root-relative (`/about/`), 1 is absolute (`https://www.sparrow.clinic/faq/` — set explicitly in front matter during FAQ creation). The site-wide canonical generator uses an empty `site.url`. **Set `site.url = "https://www.sparrow.clinic"` in `_config.yml`** and canonicals (plus sitemap URLs, plus og:url) become absolute everywhere automatically.

**robots.txt** — see Inventory section. Decision point flagged below.

**og:image** — `_config.yml` defaults to `/assets/images/og-default.jpg`, **but that file does not exist in `assets/images/`**. Every page's `<meta property="og:image">` points to a 404. Affects social previews and some LLM crawlers that use og:image as a visual signal.

**Mobile / Core Web Vitals (signals visible from the build):**
- All pages are responsive (CSS uses flex/grid with breakpoints).
- LCP candidate: hero background image `/assets/images/hero-sky.jpg` — could be `loading="eager"` and `fetchpriority="high"` hinted; currently set as CSS background, so the hint doesn't apply directly. Hero image is ~not preloaded.
- Render-blocking: Google Fonts CSS imported via standard `<link>`, no `preconnect` (actually `preconnect` IS present — good). Inter, Space Grotesk, JetBrains Mono — three families. Acceptable.
- One iframe on `/contact/` (Maps) has `loading="lazy"` ✓. One iframe on `/book/` (Zoom scheduler) does not — could be `loading="lazy"`.
- One video embed (`/about/`, `/services/`) — checked via `_includes/video-embed.html`, lazy-loading status worth confirming.
- No render-blocking JS beyond the small inline menu-toggle script.
